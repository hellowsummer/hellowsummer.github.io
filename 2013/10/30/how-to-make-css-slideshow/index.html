General Design for Slide Show

Each slide will be same height and same width as the browser window. Suppose we want a slide show with 4 slides, the total width will be 400%. Thinking the slides as a whole, move to prev/next is actually moving the position this one whole slide. The part that aligns with the browser is the part we see.

To move this one large slide, we’ll only consider the position of its left bar. CSS3 Transition property is used to achieve this “moving” process.

Prev/Next Control

To represent which slide is currently on, we use a group of radio button. Some benefits of using radio buttons:

  1. For one group of radio button, only one could be checked. This corresponds to only one slide will be shown.
  2. One radio button must be checked. This corresponds to one slide must be shown.
  3. We can autofocus on the first radio button when load the page, then use left and right arrow on keyboard to contol the selection. This would save some work to implement keyboard interaction. Instead of right arrow -> show next slide, we now have right arrow -> select next radio button -> show next slide, the first part is now automated, and the second part is achieved as part of main implementation so we don’t need any extra work.

Also notice that when we click on a label “for” the radio button, we checked the radio button, no matter the position of the two. Thus, for mouse clicking, we implement as “click on a label for the radio button”. Consider slide 2 in above diagram, we want the left triangle takes us to slide 1, so it should be a label “for” radio button 1; we ant the right triangle takes us to slide 3, so it should be a label “for” radio button 3

Note when slide 2 is shown, label for radio button 3 is shown as triangle on the right; when slide 4 is shown, label for radio button 4 is shown as triangle on the left. This requires CSS selector :checked (for radio buttons) and tilde ~ (which is called “general sibling combination selector”, used for selecting elements NOT immediately follows).

The “Progress Bar”

In this implementation, the progress bar is shown as the four rectangles at the bottom. To make it easy, the original radio buttons are keep invisible, and the rectangles are in fact four labels for radio buttons (see above notes about mouse control). This is just one way to implement this. We can also make the radio buttons visible, and directly apply style to them. Or instead of make the “current” label different, we can make previous up to current visible, and the rest invisible (or dim), to make it more like “progress bar”.

Specification on each slide.

  1. To set solid one colour background, set width: /*full width*/; height: 100%; background-color: /* some colour */
  2. For more smooth transition, use Transition on opacity
  3. Individual slides are within the one slide “container”, so each slide should has width:25%; (suppose 4 slides total)

The Left and Right Triangle

We used pure css to “draw” the left and right triangle, which uses the fact that borders meet each other at angles.

When we set an element with width: 0; height: 0;border: 1px solid, its borders looks like this:

Suppose we want a right-triangle, we need border-left to be solid; to have the angle, we must also has top and bottom border, which are set to transparent to make them invisible.

For animation of pure-css triangles, see Reference #2

Responsive Web Design

On mobile devies, we want the slides scroll up/down instead of left/right. To achieve this, several changes are to be made:

  1. change the progress bar to the right
  2. change the arrow to up and right and position at the corresponding location
  3. the width should be 100% and height should be 400% since the slides are positioned top/bottom
  4. the transitions on “left” should now be “top”
    The above changes are made for screen width less than 480px, i.e. @media screen and (max-width: 480px)
